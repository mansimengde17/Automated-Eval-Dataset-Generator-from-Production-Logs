The eval suite that grows itself

Automated Eval Dataset Generator from Production Logs

A pipeline that mines production logs, finds the edge cases and failures, and turns them into labeled test cases automatically. The eval suite grows every night from real traffic instead of going stale by hand.

Why this system exists

The operational problem it solves, and why teams keep rebuilding it by hand until someone gives them the tooling.

Why datasets are the real bottleneck

The hard part of evaluation is not the harness, it is the dataset. Hand curated golden sets are expensive to build and go stale the moment traffic shifts. This solves the data supply problem instead of the plumbing problem.

Why it lands interviews

It is a force multiplier that AI teams dream about: production traffic in, deduplicated and labeled test cases out, with humans reviewing only the labels the judges could not agree on.

What it turns into eval data

Retries after a bad answer, thumbs down signals, novel requests that do not cluster, and prompt injection attempts. The messy, valuable tail of real traffic becomes test cases.

Why the labels can be trusted

Each candidate is scored by multiple judge runs. Agreement means it is added automatically; disagreement routes to a human, and a cosine check keeps near duplicates out.

The exact workflow

Every request or task travels this path. The strip lights up in sequence so you can see the order the stages fire in.

01
Ingest

Logs are validated, PII is redacted with regex rules, and exact duplicates are dropped.

02
Sample

Uniform, stratified, or signal boosted sampling oversamples the failures and retries.

03
Cluster

Prompts are embedded and grouped by topic; points that do not cluster are outliers.

04
Label

Multiple judge runs assign quality, difficulty, expected behavior, and a golden reference.

05
Route

High agreement labels go straight in; low agreement labels land in a human review queue.

06
Detect

The eval runner scores each model and flags regressions against the previous run.

How it works in depth

The mechanisms behind each stage, and the design decisions that make the system behave well under real load.

1
Ingestion and redaction

Adapters normalize logs into one schema, regex rules strip emails, cards, and phone numbers before storage, and exact duplicates are removed by content hash.

2
Three sampling strategies

Uniform for baseline diversity, stratified by feature and model for coverage, and signal boosted to oversample negative feedback, retries, and slow calls, because those are the valuable candidates.

3
Clustering and outlier detection

Bag of words embeddings and a greedy density pass reveal the natural topics of your traffic. Points that never join a cluster are novel requests, exactly the cases a golden set is missing.

4
Multi run auto labeling

Each candidate is scored by several judge runs for quality, tagged by difficulty from simple to adversarial, given an expected behavior, and paired with a reference answer or a rubric.

5
Confidence routing and dedup

Labels the judges agree on are added automatically; the rest go to a human queue. A cosine check at 0.92 stops near duplicate cases from bloating the suite, and coverage metrics steer generation to sparse topics.

6
Eval runner and regression tracker

The harness scores any model with per category and per difficulty pass rates, then compares runs to surface new failures, new passes, and category level shifts.

How to run it

Runs fully offline with deterministic providers, so the demo and the tests reproduce byte for byte. Add API keys to switch to live models.

# offline: two pipeline nights plus a caught regression
pip install -r requirements.txt
python demo.py
# -> clusters found, cases auto labeled, degraded model flagged

# serve the generator
uvicorn evalgen.api:app --app-dir src --port 8000
curl -s -X POST localhost:8000/v1/pipeline/run -d '{"sample_target": 200}'
